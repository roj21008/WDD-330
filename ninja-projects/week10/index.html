VALIDATING FORMS

  • JavaScript validation is coded using JavaScript. This validation is fully customizable, but you must create the whole thing (or use a library).
  • If the data entered in a form field follows all the rules specified by the above attributes, it is considered valid. If not, it is considered invalid.
  • The presence of the required attribute on any element that supports this attribute means that the element matches the :requiredpseudoclass, whether or not it has a value.
  • You should use JavaScript if you want to control the appearance of native error messages or to deal with legacy browsers that don't support HTML's built-in form validation.
  • There is no standard way to change its appearance with CSS.
  • Depend on browser locale, which means you can have a page in one language but an error message is displayed in another language,
  • You must determine how to validate your data: string operations, type conversion, regular expressions, etc. Up to you.
  • To reduce user frustration, it is very important to provide as much useful information as possible to guide them in correcting their entries. You should offer hints up front so they know what to expect, as well as clear error messages.


FETCH API

  • Provides a global fetch() method that provides an easy and logical way to asynchronously fetch resources over the network.
  • The Response object, in turn, does not directly contain the actual JSON response body, but rather is a representation of the full HTTP response.
  • To make browsers send a request with included credentials in same-origin and cross-origin calls, add credentials: 'include' to the init object you pass to the fetch() method.
  • A fetch() promise will be rejected with a TypeError when a network error is encountered or CORS is misconfigured on the server side
  • All header methods throw a TypeError if a header name is used that is not a valid HTTP header name.
